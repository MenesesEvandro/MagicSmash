# Roadmap

Ideas for where Magic Smash could go next. None of this is promised or scheduled — it's a starting point for discussion, and a place to point contributors who want to help but aren't sure where to start. If one of these sounds fun to build, open an issue first so we can talk it through, then see [Contributing](README.md#contributing) in the README.

## Multiple children

- **Named profiles.** `data.profile` today holds a single name. Families with more than one toddler probably want separate profiles — name, favourite theme, and stats kept independent per child, switched the same way themes are switched now. Everything stays local; this is just structuring the existing data model to hold a list instead of one entry.
- **Export / import stats and settings** as a downloaded `.json` file, so a profile can move to a second device (a grandparent's tablet, say) without a server in between.

## Accessibility

- **High-contrast mode**, as an alternative to light/dark.
- **A more thorough `prefers-reduced-motion` story** — today it shortens animation durations; it could also cut down the number of particles spawned per keypress for kids sensitive to visual clutter.
- **A parent gate** on the settings/exit controls — something like a held two-finger long-press or a short sequence — so a toddler can't accidentally change settings or back out of the app. Not a real password (there's nothing worth protecting behind it), just enough friction that a two-year-old's hands don't trigger it by accident.
- **A screen-pinning how-to**, in Settings. No PWA can lock the OS's own exit gestures — a swipe up or the home button always gets a child out to the home screen, whatever the parent gate above stops. iOS's Guided Access and Android's Screen Pinning both genuinely lock the device to one app, but only a parent can turn either on, from the OS's own settings, before handing the tablet over — nothing on a web page can flip that switch. A short panel with the couple of steps for each platform is the honest complement to the parent gate: the gate stops a curious tap from opening Settings inside the app; screen pinning stops a swipe from leaving the app entirely.
- **Dead zones for the edges of the screen.** A toddler often holds a tablet by its edges, and those grip points land on the screen — `pressPointer()` reacts to a thumb bracing the tablet the same way it reacts to an intentional tap in the middle. An opt-in setting could ignore pointer events within a margin of the screen's edges and corners, so only touches nearer the centre of `#playArea` register. Easy to get the margin wrong in either direction (too wide and a real corner-tap goes unanswered, too narrow and grip touches still slip through), so this should ship as something a parent can turn off if it doesn't fit their kid's hands, not a fixed rule.
- **Ignoring a forearm or a flat palm resting on the screen.** The dead-zone idea above only helps at the screen's edges — it does nothing for a baby who leans an elbow or a whole forearm across the middle of the tablet while reaching with the other hand. `pressPointer()` fires `triggerInteraction()` for every `pointerdown` today regardless of how big the contact was; it never reads the `width`/`height` a `PointerEvent` already reports for its contact ellipse (a fingertip is a few millimetres wide, a forearm is not). Comparing that against a size threshold and silently dropping anything past it — no effect, no sound, no error — would let a resting arm go unnoticed while the other hand's real taps keep registering normally, since each `pointerId` carries its own contact size independent of any other pointer currently down. Simpler than it sounds: unlike the whole-hand "Super Smash" idea below, this doesn't need to track the *set* of active pointers at all, just look at one event's own size in isolation.
- **Single-switch accessibility support.** A child with limited fine motor control might play through one external switch — wired in over Bluetooth or a headphone jack, common in occupational therapy — standing in for a single spacebar press or a single primary click. That input already reaches the app today through the exact same `keydown`/`pointerdown` handlers everything else uses, so basic play already works with zero changes. What's missing is the reward matching the effort: every keypress today lands its effect at a padded-but-still-random point via `randomEffectPoint()`, the same modest single-point effect no matter which key was pressed. A browser can't tell a real spacebar tap from a switch device pretending to be one — there's no API for that distinction — so this would need to be its own opt-in setting, alongside the other accessibility toggles here, turning every press of the designated key into the whole-hand "Super Effect" already proposed below instead of an ordinary tap: a guaranteed, centred, maximal reward every time, not a matter of where the random point happens to land.

## More to press and see

- **More themes** — safari, robots, garden/insects, seasons. Same pattern as the existing ones: an icon set, a color palette, a sound, and a spot in both theme pickers.
- **An on-screen keyboard that lights up** the physical key just pressed, to help a toddler connect the key under their finger with what happened on screen.
- **A "clean the glass" reveal mode.** The screen starts fogged over — a frosted overlay, or virtual leaves — and dragging a finger or the mouse wipes it clear, revealing the current theme's colours and icons underneath. It's the touch-trail idea in Sensory & learning expansion below, put toward a visual goal instead of just a sparkle trail, which gives the dragging motion something concrete to aim for.
- **Procedurally generated splats, for the First Drawing mode below.** Every effect the app spawns today is one of a small fixed set of emoji, so the same dozen glyphs repeat all session regardless of how the touch itself looked. Generating a small closed SVG path at touch time instead — a handful of randomized control points around a base radius, closed into a blob or a rough star — would mean no two touches ever look quite identical, and would give the accumulated picture in the First Drawing mode below a more organic, painterly texture than a wall of repeated emoji. Fits alongside emoji rather than replacing them outright — some themes' character (Music's ♫, Dinosaurs' 👣) is the symbol itself, and a random blob wouldn't read the same way.

## Toddler-shaped interaction

- **"Super Smash" for a whole-hand slap.** Toddlers often hit the screen with a flat palm instead of a fingertip, landing several touches at once — `pressPointer()` treats every touch independently today, and nothing counts how many fingers are down at the same moment. Worth being precise about: that's not quite what `navigator.maxTouchPoints` tells you — it reports what the hardware can physically sense, not what's touching it right now. Actually detecting a simultaneous slap means tracking the set of active pointer IDs across `pointerdown`/`pointerup`/`pointercancel` on `#playArea` (nothing listens for the last two today). Past some threshold — four fingers at once, say — a "Super Effect" could fire instead of an ordinary tap: a screen-wide rainbow burst, a gong, a laugh. Rewards the chaos instead of just processing it as several unrelated taps.
- **An idle/attract mode.** A toddler wanders off with the tablet still open, or just pauses to stare — nothing currently changes either way. After a stretch with no keyboard, mouse, or touch input (30–60 seconds, say), the background could shift into a livelier idle state — extra shooting stars, floating bubbles, slow shapes drifting on their own — building on the ambient drifting already there via `createMagicBackground()` and `startBackgroundShuffle()`, just turned up. The next touch or keypress would "pop" the idle state back to normal play, doubling as a gentle invitation back in rather than a plain screensaver.
- **Turning a held physical key into one continuous effect, not a machine gun.** A toddler leaning on a laptop keyboard triggers the OS's own key-repeat: `keydown` fires again and again for as long as the key stays down, and `pressKey()` never checks `event.repeat` — every one of those repeats runs a full `triggerInteraction()`, which likely means hundreds of new DOM particles a second for one key held for two. Throttling the extras away would fix the overload but throw away something a toddler would probably enjoy more: recognizing a held key (`event.repeat === true` marks every repeat after the first) and turning it into a single continuous effect that grows for as long as the key stays down — a laser that lengthens, a note that sustains, a balloon that inflates — resolving only on `keyup`, which nothing listens for today either. Needs the same kind of per-frame growth loop as the behavioural themes below, so it shares groundwork with that rather than starting from nothing.

## Play modes shaped by playstyle (instead of age)

While some of these ideas naturally fit older or younger toddlers, labeling UI options with strict age ranges (e.g., "1–2 years") can frustrate children and limit exploration. Instead, the app's alternative play modes — living safely behind the parent gate — should be grouped by the type of play or "mood," letting the parent choose based on what the child needs at that exact moment:

- **Discovery (Free Play):** Pure cause and effect with no right or wrong answers, rewarding big, chaotic interactions. *Includes the "Peek-a-boo" mechanic, "Super Smash" detection, and "Bubble Wrap".*
- **Focus & Logic:** Modes that ask for intentional movement, working memory, or listening. *Includes the "Where's the lion?" voice prompts and the "Floor Piano" spatial mapping.*
- **Creative (Art):** Modes where the chaos stops evaporating and starts accumulating into something permanent. *Includes the "First Drawing" canvas and "Magic Shape" recognition.*
- **Sensory (Calm):** Passive, low-stimulation experiences designed for winding down or sensory overload relief. *Includes the "Night Light" breathing screen and time-aware Bedtime transitions.*

- **A "Peek-a-boo" mode (cause and effect), for roughly ages 1–2.** The whole-hand-slap phase, before aimed taps exist. The mechanic is a two-beat loop: the child hits the screen — anywhere, with anything — and a box appears; they hit it again — anywhere, it doesn't matter where — and an animal pops out. Then it starts over. There is deliberately no right or wrong answer and nothing to aim at: at this age the discovery *is* the game — my action makes something happen on the screen — and the reward has to be immediate and unconditional. `pressPointer()` already treats every touch identically with no concept of a target, so this mode is less about adding logic than about removing it: one shared two-step state (waiting for box / waiting for pop) instead of per-key effects. The animal icon sets already in `themeIcons` (farm, ocean, dinosaurs) supply the cast, and the whole-hand "Super Smash" idea above is the same insight about the same hands — a flat palm is valid input here, not noise.
- **A "Where's the lion?" mode (memory and attention), for roughly ages 2.5–4.** A few animals appear on screen, boxes cover them, and a voice asks the child to find a specific one — "where's the lion?". That one sentence quietly asks for three different skills at once: working memory (where was the lion five seconds ago?), language comprehension (connecting the spoken word to the animal), and fine motor control (tapping *that* box, not a smash across all of them) — which is exactly why it fits an older child than everything else in the app. It leans on two other roadmap ideas: the voice comes from the `SpeechSynthesis` work in Sensory & learning expansion below, with the animal names living in the six `src/languages/*.json` files the same way the per-letter word lists would; and the lion itself would arrive with the safari theme in More to press and see above (farm's cow, horse, and pig could carry the mode until then). The one design point worth settling early: this is the first place in the app where a tap can be "wrong," and a wrong box should open onto something gently funny — a giggle, a wiggle, a "try again!" — never onto anything that reads as failure to a three-year-old.
- **A "First Drawing" mode (canvas / stamp art), for any age.** Everything a child makes today is temporary by design: each particle is a DOM element playing a fixed animation that removes itself on `animationend`, so nothing survives longer than about a second. In an Art mode, the play area's backdrop would be a `<canvas>`, and every touch would still fire its usual animation — but also stamp the shape and colour permanently onto the canvas underneath. The chaos stops evaporating and starts accumulating: ten minutes of gleeful smashing becomes a picture. Two other roadmap ideas slot straight in — the shake-to-clear gesture in Device sensors below is this mode's natural eraser (a real Etch A Sketch, finally), and the parent gate above guards the one deliberate control the mode needs: a save button. `canvas.toDataURL()` generates a `.png` of the artwork and downloads it directly on the device — the child's first digital drawing, produced and saved without a byte leaving the tablet, exactly inside the no-server rule. Worth noting for whoever builds it: this is the one place the Magic Mirror camera idea below must *not* compose into the canvas — the export should only ever contain stamps, never a video frame.
- **A "Tidying Up" mode (object permanence and order), for roughly ages 1.5–3.** Every effect today has exactly two fates: play a fixed animation and vanish within a second, or, in First Drawing above, stamp permanently and just sit there. This mode would add a third: falling and coming to rest on the floor of the play area, piling up rather than gone or frozen — needing the same kind of small per-object physics the behavioural themes below would add (a position and a fall speed tracked every frame, not a fixed keyframe). A basket drawn in one corner becomes the goal: dragging a settled object into it — which needs the touch-dragging groundwork the trail-effects idea in Sensory & learning expansion below is also waiting on — tidies it away, and an empty floor triggers a burst of confetti and a "parabéns!" Toddlers already love taking things out of containers and putting them back; this is probably the most demanding single idea on this whole list, since it leans on three separate pieces of groundwork nothing here has yet: persistent objects, real per-object physics, and touch dragging.
- **A "Bubble Wrap" mode (tactile satisfaction), for any age.** The screen fills with a grid of large translucent bubbles, and each tap pops one — nothing to find, nothing to build, just the pure pleasure of popping. The bubbles being big and edge-to-edge means every touch lands on one without aiming, and a flat palm pops a whole cluster at once — the Super Smash insight above, built into the board itself instead of detected. Its state is refreshingly simple next to the other modes here: each bubble is one element with a popped flag, no physics loop, no dragging — the grid just regenerates with a cheerful animation (and, if the haptic feedback idea in More to press and see above lands, a buzz) once the last one goes. The genuinely new work is the sound: `playTone()`'s Bubbles sound today is a rising bloop — 640 Hz ramping *up* toward 1020 — which is the exact opposite of a bubble-wrap snap. A convincing pop is a tiny burst of filtered noise, not a swept oscillator tone, so this mode would introduce the first sound in the app that isn't an oscillator at all — and getting that snap *right* is most of what makes the mode satisfying, since the whole fantasy lives in the fingertip-to-ear illusion.

## Shape recognition (gesture drawing)

- **A "Magic Shape" mode.** Pointer Events already report a finger's live path as it moves, but `pressPointer()` only follows that path for the mouse today (`event.type === "pointermove" && event.pointerType === "mouse"`) — a finger dragged across the screen registers just its initial tap, the same gap the touch-trail idea in Sensory & learning expansion below needs closed first. Once a real path is being tracked, that same data could feed a small offline gesture recognizer — the `$1 Unistroke Recognizer` is a well-known example, a few KB of geometry matching with no model to download and no network call — to tell a rough circle from a rough straight line. A big circle traced on the screen could summon an oversized "Super Sun" filling half the play area; a rough straight line could send a train (the Vehicles theme's 🚂) rolling all the way across the screen — a bigger, one-off spectacle rather than the usual localized effect that fades within a second, closer in spirit to the whole-hand "Super Smash" reward above than to an ordinary tap. The real design problem isn't the recognition itself, it's telling an intentional shape from an accidental scribble — a minimum stroke length and a generous fit tolerance would matter more here than recognizer accuracy, so a toddler's wobbly attempt at a circle still counts as one, while a random scrawl matches nothing at all.

## Playing together

- **Multi-touch bonding.** A modern touchscreen tracks ten or more simultaneous contacts, but the app treats every touch today as its own independent, momentary tap — nothing links one finger's contact to another's, and nothing keeps a finger "down" and tracked past its initial `pointerdown` (the same gap the whole-hand "Super Smash" idea above needs closed: tracking active pointer IDs across `pointerdown`/`pointerup`/`pointercancel`, none of which `pressPointer()` currently follows through on). With that tracking in place, a parent holding one finger down in a corner of the screen could become an anchor: every separate touch the child makes anywhere else draws a live beam back to it — a rainbow arc or a trail of stars, drawn with SVG or on the same `<canvas>` the First Drawing mode above would add — visibly linking the child's finger to the parent's. It turns a solitary smash into something the two of them are visibly doing together, on the same screen, at the same time.
- **A "Magnet" mode (attraction and repulsion).** The physics sibling of the bonding beam above: the same held finger, but instead of drawing a beam of light to the other hand's touches, it pulls their particles in. Every effect today flies outward on its own fixed keyframe and dies where it lands; here, a finger held anywhere on the screen becomes a magnetic pole, and each tap from the other hand spawns particles that swarm toward it — or scatter away from it — trailing after the held finger as it slides around. It needs exactly the two pieces of groundwork already named elsewhere in this document, and nothing new: the held-pointer tracking the bonding idea above depends on, and the per-frame physics loop from Behavioural themes below (a particle steering toward a moving target every frame is the same kind of loop as one drifting with tilt gravity). Works as a two-person game — a parent holds the pole, the child feeds it — but just as well solo, one small hand holding still while the other one smashes, which for a toddler is its own little coordination exercise.

## Behavioural themes (variable physics)

- **Let a theme change how particles move, not just how they look.** Every theme's effect in `makeThemeMechanic()` already gets its own `@keyframes` animation — `beachWave`, `spaceFloat`, `oceanDrift`, and the rest all differ in direction and easing — but every one of them is the same shape underneath: a fixed, one-shot animation that plays for under a second and removes itself on `animationend`. None of them actually know where they are or react to anything. Ocean bubbles that drift gently upward and pop only once they reach the top, Space debris that bounces off the screen edges instead of fading out, Weather snowflakes (already in that theme's icon set) that sway side to side while slowly falling — none of that fits a fixed keyframe, because each depends on the particle's actual position over time. It would need at least those themes' effects to run their own small `requestAnimationFrame` loop instead of firing a CSS animation and walking away. More work than adding a theme today (icon set, palette, sound, one keyframe), but Ocean, Space, and Weather already exist and are exactly the themes these three ideas fit — "winter" from the seasons idea above could ride on Weather's existing snowflake rather than needing a theme of its own.

## Whole-screen visual modes

- **A "Kaleidoscope" mode (mirrored symmetry).** Every touch today spawns exactly one effect at one point — `makeSparkles()` and `makeThemeMechanic()` both take a single `{clientX, clientY}` and fire once. Mirroring that point across a handful of fixed axes through the centre of `#playArea` — 4-way or 8-way symmetry, simple reflection math, no trigonometry beyond a sign flip and a subtraction — and calling the same spawn functions once per reflection would turn one tap into a small radial pattern, and a few taps in a row into something that starts to look like a real kaleidoscope. Lighter than it sounds next to the other big ideas here: it needs no new physics loop, no new state, no new API — just calling code the app already has, several times, at coordinates worked out at the moment of the tap.
- **A liquid, rippling background.** The background today is `createMagicBackground()`'s icons drifting on fixed CSS animations — nothing reacts to a touch happening near it. A water-surface feel, an expanding ring that visibly bends the colour and icons it passes under before bouncing once off the play area's edges and fading, is a heavier problem than anything else proposed as a "theme" so far. An SVG `<feDisplacementMap>` filter driven by an expanding radial gradient is the lighter-weight route — distorting what's already there without redrawing it — where a full `<canvas>` simulation reading back pixel data is the more literal, more expensive one. Either way this sits in the same territory as the behavioural themes above — something continuously simulated rather than a fixed keyframe — just applied to the whole backdrop instead of one particle at a time.

## Device sensors

Both of these stay local — motion and orientation readings never leave the device, so they don't conflict with the no-data-leaves-the-device rule. The main thing to design around is permission: iOS requires an explicit `requestPermission()` call tied to a user gesture (a settings toggle, not something that can just start listening on page load), while most other browsers don't ask at all.

- **Shake to clear**, Etch A Sketch style. `DeviceMotion` can detect a shake past some threshold, and a "swoosh" sound plus a clear of everything currently on screen — the background objects `createMagicBackground()` maintains, plus any sparkles or theme effects still mid-animation — would give a satisfying, deliberate way to wipe the slate, distinct from ending the session entirely. Needs a debounce so normal handling (picking the tablet up, walking with it) doesn't trigger it by accident, the same concern behind the dead-zones idea above.
- **Dynamic gravity.** `DeviceOrientation` reports the tablet's tilt; particles could drift toward whichever edge is now "downhill" when a child tilts the device, turning cause and effect (I tilt, things move) into something they can feel out for themselves. This needs the same kind of per-frame physics loop as the behavioural themes above, rather than fixed keyframes, so the two ideas share their groundwork — building one probably means building most of the other.
- **Reacting to a dark room.** `AmbientLightSensor` can report the room's actual light level, letting the game notice a child being carried into a dim bedroom and soften itself in response — dimming particle brightness, easing the background toward much darker tones — without a parent touching a setting. Worth flagging early: support is thin, and Safari has never shipped it at all, which matters a lot for an app that mostly runs on iPads. A `Date`-based fallback makes the gap survivable — the same clock read the time-aware themes idea in Routine awareness below already proposes, reused here as a fallback path rather than the whole mechanism.

## Camera and microphone, kept entirely local

Both of these need `getUserMedia` — a genuinely sensitive-feeling permission prompt, camera or microphone, even though everything below stays on-device (a local `<video>` element, or an `AnalyserNode` reading live amplitude) with nothing recorded, stored, or transmitted anywhere. That distinction is worth being careful about in how it's presented, not just in how it's built: opt-in only, never requested automatically, explained in Settings in plain language before the browser's own permission prompt ever appears, so a parent knows exactly why a keyboard-smashing app wants their camera or microphone before they're asked to grant it.

- **Magic Mirror.** The front camera's live feed, blurred or dimmed with CSS, as the play area's background instead of the usual gradient — so the particles, bubbles, and stars a keypress spawns explode over the child's own reflection, which toddlers tend to find endlessly fascinating on its own. The video stays in a local `<video>` element the whole time; nothing is ever captured to a canvas, saved, or sent anywhere, and the camera turns off the moment the toggle does.
- **Responding to laughs and claps.** An `AnalyserNode` reading the microphone's live volume, entirely on-device, could trigger a burst of colour on a loud spike — a laugh, a clap, an excited shout — turning the child's own noise into cause and effect, not just their touch. The hard part isn't the API, it's the threshold: a loud spike doesn't know a delighted shriek from a dog barking or the TV, so this would need real tuning (and an easy off switch) to avoid firing on everything but what it's meant for.
- **Detecting a blow, not just a spike.** The laughs-and-claps idea above already proposes an `AnalyserNode` reading the microphone's live amplitude for a sharp, loud spike; a blow is closer to the opposite signature — a sustained, low-frequency rush of noise rather than one loud instant — so it would need its own detector (sustained energy in the low bins over roughly a second, say) rather than reusing the same spike threshold. Once that's tuned, it fits two things that already exist: the Bubbles theme, where a blow could send a fresh handful of bubbles floating up the screen, and the "clean the glass" reveal mode above, where blowing — rather than a dragged finger — could be what clears the fog, closer to how a toddler would actually try to clean a real window.

## Sensory & learning expansion

- **Voice and pronunciation, as an alternative to sound.** The browser's built-in speech synthesis (`SpeechSynthesis`) runs locally on most platforms with no network call, so it fits the offline-first rule. Toggled in Settings in place of the synthesized tone — never both at once, the same way sound is on/off today — a friendly voice would say the letter or a short, toddler-friendly word for it in the selected language ("A is for Apple," or whatever reads best to a two-year-old in that language; a literal translation of the English word usually won't start with the same letter, so this needs its own small per-letter word list per language, not a translation of one). Voice mode would only cover letters and numbers — every other key (space, arrows, punctuation) has no natural word to say, so it would keep the existing tone or just stay quiet. The six `src/languages/*.json` files already give every language its own home for this list; no new architecture needed, just content.
- **Colour tied to note.** Every theme's tone is already quantized to a shared pentatonic scale (`nearestPentatonicNote()` in `playTone()`), so every sound a key makes is already one of a small, fixed set of notes. Each of those notes could also be tied to one fixed colour, always — C always red, D always orange, E always yellow, and so on — regardless of which theme is playing or which icon happens to show up. Repeated across hundreds of taps a session, that consistent pairing gives a toddler a passive, wordless way to start associating a sound with a colour, no lesson involved, just repetition doing the work.
- **Panning sound to where the touch landed.** `playTone()` wires its oscillator straight into a `GainNode` and then `audioContext.destination` — nothing in that chain knows or cares where on the screen a touch happened, so every tap sounds identical in both speakers or ears no matter where it lands. Adding a `StereoPannerNode` between the gain and the destination, with its `pan` value derived from the touch's `clientX` relative to `#playArea`'s width (−1 at the left edge, 0 in the middle, 1 at the right), would mean a tap in the corner plays from that corner — a spatial companion to the colour-tied-to-note idea above rather than a replacement for it, gently teaching where-I-touch-is-where-it-comes-from without a word of explanation. Keyboard presses have no natural X position to pan from (`triggerInteraction()` already picks a random point for those, per its own doc comment), so this would only really apply to pointer taps; keyboard play would just stay centred.
- **A floor piano: position fixes the note, not chance.** The two ideas just above already point at the same destination from different directions — colour tied to note gives every pentatonic note a fixed colour, and panning ties left/right sound to touch position — without either one making position *determine* pitch. Dividing `#playArea` into a handful of tall vertical bands, one per pentatonic note, and having `playTone()` read which band a touch landed in instead of picking randomly would close that gap directly: the leftmost band is always the same note, always the same colour, wherever inside it the taps land. It turns the whole screen into a real instrument a child can learn by feel — hit the red stripe, always get that note — rather than a novelty that happens to sound nice.
- **Rhythmic echo (call and response).** `state.lastPointerTime` already tracks the gap between one touch and the next, but only to debounce the mouse trail — nothing keeps the actual sequence of gaps around. Recording a short rolling buffer of tap timestamps instead, and noticing a pause past some threshold (the child stopping, waiting), could let the game play the same rhythm back — tap, tap, tap-tap — as lights and pentatonic tones through `playTone()`, which are already quantized to that same scale, rather than a random pitch each time. The tricky part is exactly the same tension the idle/attract mode above has to navigate, just on a much shorter timescale: telling "the child paused, waiting for their echo" from "the child just stopped to look at something else" is a threshold to tune, not a fact to detect. Gets a child their first wordless taste of turn-taking and imitation — I go, then you go — without either side saying a thing.
- **A second language, alongside the first.** Once voice and pronunciation above exists, alternating between two active languages instead of one — the same key saying "Maçã" in Portuguese on one touch and "Manzana" in Spanish, spoken natively, on the next — would turn every letter into a small, repeated introduction to a second language, not just a pronunciation aid for the one already selected. `data.language` is a single value today (`activeLanguage()` always picks exactly one); this needs it to become a short ordered list instead — the same shape of change as giving `data.profile` room for more than one child, above.
- **Trail effects for touch, not just the mouse.** Dragging already spawns a pointer trail in `pressPointer()` — but only for mouse movement; a finger dragged across a touchscreen today only registers the initial tap, since touch `pointermove` is explicitly skipped. Extending that to touch would let a toddler who doesn't yet have precise, aimed-tap coordination paint a trail of sparkles — or slide through a musical scale, xylophone-style — just by dragging a finger around, no aiming required.
- **Trail particles that feel the drag's speed, not just its position.** `makePointerTrail()` already knows where the pointer is on each move; it has no idea how fast it got there — every trail spark today is the same fixed random size regardless of speed. Tracking the time and distance between one `pointermove` and the next (`state.lastPointerTime` already tracks half of that) would let a slow, lazy drag spawn big unhurried bubbles and a fast, excited one spawn thin sparks that vanish quickly — the same trail, responding to how the child is actually moving. Works for the mouse trail that already exists today, and would carry over automatically once touch dragging above lands too.

## A gentler end to playtime

- **Wind down instead of cutting off.** Today `tick()` calls `endGame()` the instant the clock hits zero — the key stage disappears and the recap dialog pops up mid-keystroke, whatever the child was doing. For a longer session (15, 30 minutes) that's a rough way to end. Instead, the last stretch could wind down: the screen dims smoothly, a lullaby plays, and keys stop producing new effects — signalling that playtime is ending before it actually stops, so a toddler has a moment to notice instead of the screen just changing all at once. The Bedtime theme already has most of the raw material for this (a calm sine tone, the moon/star/sheep icon set, a quiet colour palette) — this would lean on that rather than build a new mood from scratch.
- **A parent-set limit, behind the parent gate.** Once that gate exists (see Accessibility above), a longer duration option is exactly the kind of setting that belongs behind it — unlike the quick 3/5/10-minute picker today, which any curious toddler in Settings can already change themselves.
- **A passive "night light" mode.** A screen that just breathes: warm colours (amber, orange, red) pulsing on a slow, calm rhythm, meant to be looked at rather than played with, any touch stirring only the gentlest ripple of colour rather than a sound or a burst. Unlike most of the groundwork this document asks for, one piece of this already exists and ships today — `keepScreenAwake()` keeps the screen from dimming and locking mid-session, exactly what a night light needs to stay lit unattended. The rest leans on what's already proposed nearby: the Bedtime theme's existing calm palette and tone as a starting point, and the dark-room detection in Device sensors above as a natural, though not required, way in — a room going dark is as good a cue for "night light time" as it is for softening whatever theme is already playing.

## Routine awareness

- **Time-aware themes.** `data.theme` is picked by hand today — the quick picker on the welcome screen, or Settings — and stays whatever it was last set to no matter when the app is opened; `initializeApp()` only ever overrides it to fall back to a theme that still exists (`initialData.theme`, if the saved one was removed), never based on the clock. A native `Date` read at startup could change that: opened after, say, 18:30, the app could boot straight into the Bedtime theme — its calm palette, moon/star/sheep icons, and quiet sine tone already exist — instead of a parent having to remember to switch it manually before handing the tablet over near sleep time. That saves a parent from accidentally launching something loud and stimulating right before bed. Worth being honest about a gap this would expose: the "reduced volume" part isn't just a theme swap — sound today is only ever on or off (`data.sound`, a boolean; see `updateSound()`), with no continuous level in between, so a real volume dip would need an actual gain control to turn down first, not just a theme change riding on one that doesn't exist yet. Should stay a suggestion the app makes, not a silent override — a parent picking a livelier theme on purpose at 9pm on a Saturday shouldn't get overruled by the clock without at least seeing why.
- **App shortcuts for specific modes.** `manifest.webmanifest` today defines nothing but icons, colours, and a single `start_url` — no `shortcuts` array, the manifest field that lets a long-press on the home-screen icon pop up a small native menu of alternate entry points. Adding a few — Bedtime, Magic Mirror, First Drawing — each pointing at `index.html` with a query parameter the app reads on boot (`initializeApp()` only ever reads persisted `data` today, never the URL) would let a parent jump straight into the mode they want before handing the tablet over, instead of opening the app cold and digging through Settings with a toddler already reaching for it. The manual, on-demand counterpart to the automatic clock-based switch time-aware themes above proposes — a parent overriding by hand exactly what the clock would otherwise guess.

## Explicitly not planned

Some things will probably never be added, on purpose:

- Analytics, crash reporting, or any telemetry.
- Cloud sync or accounts of any kind.
- Ads, in-app purchases, or third-party SDKs.
- Anything that requires an internet connection to function.

If a proposed feature needs a network call to work, it's the wrong feature for this app — see the "one firm rule" in [Contributing](README.md#contributing).
